# ファントム仮想企業シミュレーション

> このリポジトリは、架空の仮想企業「ファントム」をAIでシミュレートする実験用プロジェクトです。
> 登場する会社・社員・案件・ログはシミュレーション上の架空設定です。

## ダッシュボードの公開方針

`DASHBOARD.md` と `docs/index.html` は、仮想企業シミュレーションの現在状況を確認するためのダッシュボードです。
publicリポジトリやGitHub Pagesで公開する場合は、公開前にSecrets、個人情報、実データが混入していないことを確認してください。

GitHub Pagesで公開する場合は、`docs/index.html` を表示対象にできます。


このリポジトリは、GitHub上のMarkdownファイルを会社の記憶・台帳として使い、GitHub ActionsとOpenAI APIで仮想企業「ファントム」を自動進行させるMVPです。PCの電源が落ちていても、GitHub Actionsが4時間ごとに社員3人分の行動をAIに決めさせ、状態と履歴をMarkdownへ保存します。

## シミュレーション概要

- 会社名は **ファントム** で固定です。
- 正式名称と法人形態は初期状態では未定です。
- 法人形態は、起業前フェーズ中に代表AI「佐藤 直樹」が初期資金、信用力、営業先への見え方、設立コスト、採用・資金調達、リスク許容度、他社員の意見を踏まえて判断します。
- Day 0までに未決定の場合、起業日イベントで必ず代表が決定します。
- 事業内容は、一般企業へのAIフローシステム導入支援です。
- 会社資金と社員個人資金は分けて管理します。

## フェーズ

| フェーズ | 期間 | 内容 |
| --- | --- | --- |
| 起業前 | Day -7〜Day -1 | 法人形態検討、営業資料、見込み顧客、商品設計、体制整備 |
| 起業日 | Day 0 | 事業開始、正式名称・法人形態確定、初期サービス・営業方針確定 |
| 起業後 | Day 1以降 | 通常の事業運営、営業、案件対応、改善活動 |

## ディレクトリ構成

```text
.
├─ data/                  # 会社・社員・顧客・案件・記憶のMarkdown台帳
├─ logs/                  # logs/YYYY-MM-DD/HHMM.md のターン履歴
├─ rules/                 # シミュレーション制約
├─ prompts/               # OpenAI APIへ渡すプロンプト
├─ scripts/               # Markdown読み書き、API呼び出し、検証、状態更新
└─ .github/workflows/     # 4時間ごとのGitHub Actions
```

## 実行方法

### ローカル実行

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# .env に OPENAI_API_KEY を設定
python scripts/run_simulation.py
```

### GitHub Secrets

1. GitHubリポジトリの **Settings** を開きます。
2. **Secrets and variables** → **Actions** → **Secrets** を開きます。
3. `OPENAI_API_KEY` を追加します。

### GitHub Variables

必要に応じて **Variables** に以下を設定できます。

- `OPENAI_MODEL_NORMAL`：通常ターン用。例：`gpt-4.1-mini`
- `OPENAI_MODEL_IMPORTANT`：起業日や重要イベント用。例：`gpt-4.1`

未設定の場合は、スクリプト側のデフォルトを使用します。

### 手動実行

GitHubの **Actions** タブで `Virtual Company Simulation` を選び、**Run workflow** を押すと手動実行できます。

## 自動実行

GitHub ActionsはUTC基準のcronで、日本時間 00:17 / 04:17 / 08:17 / 12:17 / 16:17 / 20:17 に対応します。
GitHub ActionsのscheduleはUTC基準で、実行は数分遅れる場合があります。

```yaml
cron: '17 15,19,23,3,7,11 * * *'
```

## コスト節約設計

- 毎ターンAIを使いますが、AI入力は `data/current_state.md` を中心にします。
- 社員3人分の行動判断を1回のAPI呼び出しでまとめます。
- 直近ログは最大3ターン分だけ渡します。
- 各社員ファイルには長期記憶サマリを持たせます。
- `logs/` は人間用履歴として保存し、毎回全文はAIへ渡しません。
- 通常ターンは軽量モデル、重要イベント時は高性能モデルを使える構造です。
- 日次要約を `data/memory/daily_summaries.md` に追加できる構造です。

## 安全設計

- AI出力はJSONとして検証します。
- 不正JSONの場合は1回リトライし、失敗時はエラーログを残して既存Markdownを破壊しません。
- 疲労、モチベーション、評判、スキルなどは0〜100に補正します。
- 会社資金と社員個人資金は別項目で管理します。
- 個人資金から会社資金への移動は `personal_fund_to_company` イベントがある場合のみ処理します。
- 起業前フェーズの売上は原則0円に補正します。
